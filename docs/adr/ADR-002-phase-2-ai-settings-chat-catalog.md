# ADR-002：Phase 2 采用“设置 -> 对话 -> Catalog”顺序

- 状态：Accepted
- 日期：2026-04-17
- 决策范围：AI Ops Phase 2

## 1. 背景

AI Ops Phase 1A 已建立最小 `platform_action` 闭环，并形成了以下现实基础：

1. 结构化 AI 分析已经存在。
2. 受控动作执行闭环已经存在。
3. action registry 已经存在。
4. AI 运行时配置仍主要依赖环境变量。

Phase 2 计划新增三类能力：

1. AI 设置中心
2. 对话式 AI 分析页
3. Skill Catalog

本 ADR 用于固定这三项能力的优先级、边界与设计取舍。

## 2. 决策

### 2.1 为什么 Phase 2 顺序是“设置 -> 对话 -> Catalog”

决策：
Phase 2 按“设置 -> 对话 -> Catalog”的顺序实施。

原因：

1. `设置` 是控制面，`对话` 和 `Catalog` 都依赖统一的运行时配置与功能开关。
2. 若没有设置中心，对话页和 Catalog 只能继续依赖环境变量语义，无法形成产品内治理能力。
3. 对话页依赖稳定的 provider、model、baseURL、enabled 状态，否则不同 AI 入口会出现不一致行为。
4. Catalog 需要展示能力可用性、风险等级、入口类型，也依赖统一配置与状态解释。
5. 先做设置中心可以降低后续页面的集成复杂度和灰区解释成本。

### 2.2 为什么对话页不能直接执行动作

决策：
对话页只负责解释、建议与跳转，不直接执行平台动作。

原因：

1. Phase 1A 已经建立 `dry-run -> confirm -> execute -> verify` 的受控闭环，这是当前安全边界的一部分。
2. 如果聊天页直接执行动作，将绕过现有 `AIOpsAssistant.vue` 的确认机制与执行可视化流程。
3. 对话模型输出具有开放性，不能作为高风险或中风险动作的直接执行入口。
4. 当前阶段还没有审批中心，也没有适用于聊天触发的额外审查层。
5. 对话页更适合承担“解释为什么建议这么做”和“把用户导向安全执行入口”的角色。

### 2.3 为什么 Catalog 先做只读，不做编辑器

决策：
Skill Catalog 第一版只做只读能力目录，不做编辑器。

原因：

1. 当前仓库已有 action registry，可直接复用并转化为只读目录，投入低、边界清晰。
2. 编辑器意味着元数据 schema、prompt 管理、版本管理、校验、权限、发布流程，这会显著扩大范围。
3. 第一版的关键价值在于“发现能力”而不是“创建能力”。
4. 只读 Catalog 能帮助用户理解当前已有 AI 能力，而不会引入新的执行风险。
5. 在未建立 skill 生命周期管理前，编辑器会导致能力质量难以治理。

### 2.4 为什么当前不做用户可编辑 skill 平台

决策：
Phase 2 不建设用户可编辑 skill 平台。

原因：

1. 用户可编辑 skill 平台本质上是新子系统，而不是简单页面功能。
2. 它会引入 prompt 设计、参数 schema、权限控制、发布边界、审计追踪等额外问题。
3. 现阶段平台尚未建立 skill 审核与回滚机制，开放编辑将直接推高系统风险。
4. 当前最有价值的工作是把已有能力产品化，而不是把能力创作平台化。
5. 这不符合当前阶段“先收敛、后扩展”的工程策略。

## 3. 后果

### 正向影响

1. Phase 2 范围更聚焦，工程顺序明确。
2. 运行时配置语义得到统一。
3. 对话页不会破坏现有受控动作边界。
4. Catalog 可以快速落地并复用现有 action registry。

### 负向影响

1. 用户短期内不能在平台内自定义 skill。
2. 对话页第一版不具备“所说即执行”的体验。
3. Catalog 第一版更偏能力说明与跳转，而非可编排平台。

## 4. 后续计划

1. Batch 5 先完成 AI 设置中心。
2. Batch 6 完成短会话对话页。
3. Batch 7 完成只读 Skill Catalog。
4. Batch 8 完成联动与硬化。
5. 用户可编辑 skill 平台仅在未来阶段另行立项评估，不纳入当前 Phase 2。
