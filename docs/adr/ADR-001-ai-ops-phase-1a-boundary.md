# ADR-001: AI Ops Phase 1A 边界决策

- 状态：Accepted
- 日期：2026-04-15
- 决策范围：AI Ops Phase 1A

## 1. 背景

EOMS 当前 AI 能力已具备分析与建议，但不具备受控动作闭环。项目需要在安全边界内逐步引入“可执行能力”。

为降低风险，团队需要先明确动作分层与阶段边界，避免在初期引入节点资源操作与不可控行为。

## 2. 决策

### 2.1 为什么拆分 `platform_action` 与 `node_action`

决策：动作体系按执行位置与风险域拆分为两类。

原因：

1. 执行位置不同：平台内执行与节点侧执行的风险面完全不同。
2. 鉴权链路不同：平台动作可依赖现有 JWT + RBAC；节点动作需额外 agent 侧鉴权与下发链路。
3. 审计粒度不同：节点动作需要额外记录节点执行上下文。
4. 失败语义不同：节点离线、回传失败、下发超时等问题在平台动作中不存在。

### 2.2 为什么 Phase 1A 只做 `platform_action`

决策：Phase 1A 仅实现 `platform_action`。

原因：

1. 可复用现有后端与告警模块，交付路径短、可控。
2. 无需引入节点执行器改造，避免跨模块高耦合。
3. 风险可收敛在平台数据层，便于先验证状态机、dry-run、审计模型。
4. 有利于尽快形成“分析 -> dry-run -> 确认 -> 执行 -> 复检 -> 审计”闭环基线。

### 2.3 为什么首批只保留 `acknowledge_alert` 与 `create_incident_timeline_note`

决策：Phase 1A 首批动作限定为两个低风险动作。

原因：

1. 两者均为平台内数据变更，副作用可预测。
2. 均可通过结构化校验与审计完整追踪。
3. 可覆盖闭环关键路径（参数校验、权限校验、执行、复检、审计）。
4. 便于先验证框架能力，再逐步增加动作目录。

### 2.4 为什么暂不做 `trigger_metrics_recheck`

决策：暂时移出 Phase 1A。

原因：

1. 当前系统中“复检”语义更接近“平台侧重新评估已有数据”，尚未形成可验证的动作定义。
2. 若定义不清，容易与“节点侧真实重采样”混淆，造成范围漂移。
3. 在未明确“执行边界、结果可观测性、验证标准”前，不纳入首批动作更稳妥。

## 3. 后果

### 正向影响

1. Phase 1A 范围清晰，可快速推进实现。
2. 安全边界明确，避免提前触碰节点资源操作。
3. 审计与测试可聚焦最小闭环。

### 负向影响

1. Phase 1A 可交付动作数量较少。
2. 与节点操作相关的自动化收益需延后到 Phase 1B+。

## 4. 后续计划

1. 在 Phase 1A 验证闭环稳定后，再评估 `node_action` 与审批链路。
2. 对 `trigger_metrics_recheck` 形成严格定义（平台侧可验证）后再纳入后续阶段。
