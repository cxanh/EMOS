# ADR-003：Batch 7 采用“独立页 + 短会话 + 跳转执行”边界

- 状态：Accepted
- 日期：2026-04-17
- 决策范围：AI Ops Batch 7

## 1. 背景

在 AI 设置中心完成后，系统已具备统一的 AI 配置能力，但仍缺少“围绕当前运维上下文连续追问”的产品入口。

当前仓库的既有基础包括：

1. `AIAnalysis.vue` 负责结构化分析。
2. `AIOpsAssistant.vue` 负责受控动作闭环。
3. 现有动作执行边界由 `ai/v2` 与 action registry 保证。

Batch 7 需要引入新的对话式分析能力，但必须保持范围克制，不能演变为通用聊天平台。

## 2. 决策

### 2.1 为什么 Batch 7 采用“独立页 + 短会话 + 跳转执行”

决策：
Batch 7 采用独立页、短会话、推荐动作跳转执行的方案。

原因：

1. 独立页可以把多轮追问能力与 `AIAnalysis.vue` 的结构化分析职责分离。
2. 短会话足以满足当前运维分析需求，避免过早引入长期记忆与 agent 复杂度。
3. 跳转执行能够复用 `AIOpsAssistant.vue` 的 dry-run / confirm 安全边界。
4. 该方案能最大化复用现有架构，而不引入新的动作执行链路。

### 2.2 为什么不直接在聊天页执行动作

决策：
聊天页不直接执行动作，只展示推荐并跳转到 `AIOpsAssistant.vue`。

原因：

1. 现有 `ai/v2` 动作执行闭环已经形成明确安全边界。
2. 若聊天页直接执行动作，将绕过 dry-run 与 confirm。
3. 当前阶段没有审批中心，聊天页不能承担高风险执行入口。
4. 对话式输出具有开放性，必须和执行入口解耦。

### 2.3 为什么不做长期 memory / agent

决策：
Batch 7 不做长期 memory，不做 agent。

原因：

1. 当前需求是“围绕当前上下文进行短时追问”，不需要跨会话长期记忆。
2. 长期 memory 会引入额外存储、召回、清理和隐私边界问题。
3. agent 会引入自主规划与执行语义，直接突破当前阶段范围。
4. 当前最有价值的是把“分析”和“动作执行入口”连起来，而不是引入自主系统。

### 2.4 为什么不做流式输出

决策：
Batch 7 第一版不做流式输出。

原因：

1. 当前页面的核心价值是上下文化分析与结构化结论，不是流式聊天体验。
2. 流式输出会增加后端接口、前端消息状态与错误恢复复杂度。
3. 在范围受控的前提下，请求-响应式接口足以支撑第一版交付。
4. 现阶段优先保证输出结构稳定，而不是优化 token 级交互体验。

## 3. 后果

### 正向影响

1. Batch 7 范围清晰，容易按批次落地。
2. 可复用既有 `AIAnalysis.vue` 与 `AIOpsAssistant.vue` 的职责边界。
3. 对话页不会破坏现有动作安全边界。
4. 后端实现复杂度和测试面更可控。

### 负向影响

1. 第一版不具备长记忆体验。
2. 第一版不具备流式交互体验。
3. 用户仍需要跳转到动作页，而不是在聊天中直接完成操作。

## 4. 结论

Batch 7 的设计原则固定如下：

1. 独立页，不塞进 `AIAnalysis.vue`。
2. 短会话，不做长期 memory / agent。
3. 推荐动作跳转到 `AIOpsAssistant.vue`，不直接执行。
4. 第一版不做流式输出。
